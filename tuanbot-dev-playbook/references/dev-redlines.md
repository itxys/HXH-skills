# 开发执行红线清单（Tuanbot）

用于避免最近反复出现的高成本失误。触发任何一条红线，立即中止当前“乐观推进”，先修正基础条件。

## 1) 连续执行红线
- 不得把 `sessions_spawn` 当成常驻执行器。
- 不得在“阶段性汇报”后默认任务还在跑。
- 需要持续到完成的任务，必须有外层 watchdog/patrol（cron 或明确轮询机制）。

**硬规则**
1. 每次里程碑后检查 active run 是否仍在。
2. 若无 active run 且目标未完成，立即重启下一轮。
3. 对用户汇报“是否在跑”必须以实时状态为准，不能按意图回答。

## 2) E2E 前置条件红线
- 未通过前置校验，禁止宣称端到端测试结果。
- 典型误区：后端有配置 ≠ 当前前端会话已命中该配置。

**E2E preflight 必检项**
1. 进程/端口可用（如 4512/4514）。
2. 当前会话身份上下文有效（userId 或等效绑定）。
3. 前端 `settings.models` 已注入目标能力模型（如 image）。
4. 当前会话实际命中 backend model-config-center（非陈旧缓存）。

任一失败：先修复 preflight，再测功能。

## 3) 截图验收红线
- 不得提交空白图、失败态图、无关键证据图作为“成功验收”。
- 不得仅凭接口 200 代替用户可见验收图。

**聊天生图验收硬门槛**
同一聊天面板同屏可见：
1. 用户请求文本；
2. `generated-image` 回图（URL 或渲染图）。

不满足即标记无效，继续补抓。

## 4) 基础排查优先红线
卡住时不得直接堆复杂操作，必须先跑基础四步：
1. 端口/进程是否活着；
2. 会话上下文是否有效（非 default missing）；
3. 配置是否已同步到前端（不是只在后端存在）；
4. 最小链路是否可复现（先文本后生图）。

## 5) 变更与工作树红线
- 不得在脏工作树上混合提交无关改动。
- 不得把运行时数据、临时证据与核心修复混在同一提交。

**执行规则**
1. 最小修复提交（fix commit）与历史脏改隔离（stash）。
2. 每次提交必须给：commit hash、changed files、关键命令输出。
3. 若证据不足，进度冻结，不虚报。

## 6) 用户沟通红线
- 不得重复机械回复；不确定就先查再答。
- 不得把“计划”当“结果”汇报。

## 推荐落地
- 将体检命令固化为单入口脚本（例如 `test:gate:capability-healthcheck`）。
- 将截图验收条件做成脚本化断言（检测请求文本 + generated-image 关键锚点）。
- 将连续执行改为“事件驱动 + 低频巡检”双保险。
